## NEW FILES
- `AuraFitness/Profile/CSVParser.swift`: pure, dependency-free RFC-4180 CSV parser (`CSVParser.parse(_:) throws -> [[String]]`) handling quoted fields, embedded commas/newlines, and doubled-quote escaping.
- `AuraFitness/Profile/CSVArchive.swift`: `CSVArchiveBuilder` — pure per-category serializers (`workoutHistoryCSV`, `programsCSV`, `customWorkoutsCSV`, `customExercisesCSV`, `measurementsCSV`) plus `writeTempZip(_:)`, which writes the 5 CSVs to a temp folder and zips it via `NSFileCoordinator(...).coordinate(readingItemAt:options:.forUploading)` (stdlib-only zip, mirrors `DataArchiveBuilder`'s off-main-thread `Task.detached` pattern).
- `AuraFitness/Profile/DataImportService.swift`: `DataImportService` — single import entry point (`importFile(at:appState:)`) that security-scopes the picked URL, routes `.json`/`.csv`/`.zip` to the right importer, and maps parsed rows back into the real stores (`ProgramDatabase`, `UserPlanDatabase`, `ExerciseDatabase`, `appState.workoutLogs`/`measurements`) via their existing CRUD, with dedupe-by-id idempotency. Also contains `MinimalZipReader`, a small dependency-free ZIP central-directory reader (supports stored + deflate via the system `Compression` framework) used to unzip an imported CSV bundle in memory.
- `AuraFitnessTests/CSVRoundTripTests.swift`: round-trip tests covering measurements (nil vs. `0` distinction), a custom exercise, a custom program (1 workout / 2 exercises), a workout-history log (2 exercises × 2 sets), CSV escaping of a comma+quote name, and a `CSVParser` embedded-newline case.

## DEVIATIONS FROM SPEC (with justification)

1. **Zip unzip mechanism (import side).** The spec allowed two options: "reuse the `NSFileCoordinator` approach in reverse, or `Archive`-free: iterate expected inner filenames." Apple's `NSFileCoordinator` `.forUploading` trick only zips a *directory into* a zip — there is no symmetric public API that unzips a `.zip` back into a directory. I implemented `MinimalZipReader`, a small self-contained ZIP central-directory parser (handles the "stored" and "deflate" methods, decompressing via the system `Compression` framework — a first-party Apple framework, not a third-party dependency) that reads entries directly into memory. This satisfies the spec's actual requirement ("unzip... then import each inner CSV by filename → category, unknown files skipped") without inventing a new external dependency or a nonexistent stdlib API.
2. **`custom_workouts.csv` skip-counting gap.** The spec's exact required signature is `static func importCustomWorkouts(_ rows: [[String]]) -> Int` (a plain imported-count, no skip channel). When a `custom_workout` group references an unknown local `plan_id`, the group is correctly skipped per spec ("skip the group... do NOT fabricate a plan"), but because the function signature can only return one `Int`, that particular skip is not reflected in the toast's overall `skipped` count (all other skip paths — unparseable rows, unrecognized zip inner files, unsupported file types — are correctly counted). I preserved the spec's literal signature rather than widening it, since the spec calls out these signatures explicitly as must-match interfaces.
3. **No Xcode test target exists in `project.pbxproj`.** I discovered (and verified via `git grep`) that `AuraFitnessTests/PersistenceRoundTripTests.swift` — the existing test file the spec told me to mirror — is *not* referenced anywhere in `project.pbxproj`; there is no `PBXNativeTarget` for a test bundle, and CI (`.github/workflows/ci.yml`) only runs `xcodebuild build`, never `xcodebuild test`. This is a pre-existing condition, not something I introduced. Per the spec's explicit instruction, I registered the three new **app-target** Swift files in the pbxproj (PBXBuildFile/PBXFileReference/group/Sources entries), but I did not fabricate a brand-new test target/scheme to wire up `CSVRoundTripTests.swift` — that would be significant, blind scope creep beyond "add missing Sources entries for files I created," and risks producing an invalid pbxproj. Flagging this explicitly so the Tester/next stage is aware `CSVRoundTripTests.swift` (and the pre-existing `PersistenceRoundTripTests.swift`) cannot currently be executed via `xcodebuild test` until a test target is added out-of-band.
