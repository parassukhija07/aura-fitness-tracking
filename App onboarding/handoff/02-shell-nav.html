Phase 2

Shell & navigation

The frame that holds the four tabs together. This chapter covers the root component that owns routing and cross-tab settings, the full-screen overlay the Active Workout flow lives in, the resume banner, and the single most intricate piece of chrome in the app — the floating glass tab bar with its sliding indicator, scroll-collapse, swipe gestures and quick-action FAB. Every value, guard and edge case is spelled out; nothing here should be inferred.

SourceRouting & overlay live in combined/shell.jsx (AuraRoot, 46 lines). The tab bar + FAB live in combined/ui.jsx as TabBarEl (exported as window.AuraTabBar). The resume banner is rendered by combined/log.jsx. The scroll event that drives collapse is wired in Aura App.html itself.

Root routing — AuraRoot

AuraRoot is the single owner of which tab is showing and of every cross-tab setting. It is the only stateful component above the tab level; each tab is otherwise self-contained. It renders into #root and never unmounts.

State it holds

tab'log'set by tab bar / FAB / overlay exitsWhich of the four tabs renders. One of log · plan · progress · profile.
workoutOpenfalsestartWorkout / overlay exitsWhen true the Active Workout overlay replaces all tabs.
inProgressfalsestartWorkout / onMinimize / onExitA session exists but the overlay is closed → drives the resume banner.
darklocalStorage 'aura_dark' === '1'Profile (onDark)Theme. Read once on init, persisted on every change.
calStartlocalStorage 'aura_calstart' ?? 'Sun'Profile (onCalStart)Week-start day. Consumed by Log & Plan calendars.

PersistenceAll three settings are read inside the useState initializer with a try/catch that falls back to the default if storage is unavailable. Writes happen in the onDark/onCalStart/onLogStat callbacks immediately on change, also wrapped in try/catch. There is no debounce and no batch write — set state then write the key. Keys are defined in Chapter 1.
EdgeProfile is the only writer of calStart and logStat; Log and Plan/Progress only ever read them. Keep that one-way flow — don't let a tab mutate a setting it merely consumes.

Routing is a flat if ladder at the bottom of AuraRoot. Each tab gets only the props it needs — the prop surface is deliberately narrow and differs per tab. Match it exactly.

Theme enforcement — the MutationObserver

Easy to missThis has no on-screen UI but it is load-bearing. Without it, dark mode visibly breaks on the Plan and Active Workout sub-views.

Some sub-views (Plan internals, Workout sub-views) hard-code data-theme="light" on their .phone wrapper. To stop those from punching a light hole in a dark app, AuraRoot runs an effect that force-syncs the theme:

  • On [dark, tab, workoutOpen] change it computes want = dark ? 'dark' : 'light' and writes that attribute to every .phone element whose current value differs.
  • It then attaches a MutationObserver on document.body watching subtree · attributes (filter: data-theme) · childList. Any time a sub-view re-asserts its own theme or a new .phone mounts, the observer re-applies want.
  • The observer is disconnected on cleanup so it doesn't leak across re-runs.
NativeYou almost certainly won't need this — it's a workaround for sub-views that hardcode a theme in the prototype. In the target codebase, drive theme from one environment/appearance value and let every screen read it. Just don't replicate the hardcoded-light sub-views and you can drop the observer entirely.

Active Workout overlay

A running session is not a tab — it's a full-screen takeover. While workoutOpen is true, AuraRoot returns WorkoutApp early and no tab bar exists. The tab the user came from is remembered only implicitly (every exit returns to Log).

Launch — startWorkout(name, empty)

A single function opens the overlay. It sets workoutCfg = { name: name||'', empty: !!empty }, flips inProgress = true, then workoutOpen = true — in that order. WorkoutApp receives workoutName={cfg.name} and emptyMode={cfg.empty}.

empty = false

Seeded session (the day's planned "Push Day A" template loads with its exercises). Default path.

empty = true

Blank "build as you go" session — no exercises, user adds them live. See Chapter 5.

Three exits

WorkoutApp is handed three callbacks. The difference between Exit and Minimize is the whole point of the resume banner — get it exactly right.

CallbackSetsLands onResume banner after?
onExit
discard, or finish & save		workoutOpen=false · inProgress=false · tab='log'Log tabNo — session is over.
onMinimizeworkoutOpen=false · inProgress=true (kept) · tab='log'Log tabYes — banner shows.
The distinctionBoth close the overlay and land on Log. The only difference is inProgress: Exit clears it (no banner), Minimize keeps it (banner visible). Finishing and discarding both go through onExit — neither leaves a resumable session.

Entry points

Every way to start a workout funnels through startWorkout. There are three:

  • Log → "Start Workout" (today's planned card) → onStartWorkout(meta.name || 'Workout', false) — seeded.
  • Log → Add-workout sheet → "Empty Workout" → onStartWorkout('…', true) — blank mode.
  • FAB → "Start Workout" → on the Log tab this opens the add-workout source sheet (kind:'add'); on other tabs it routes to Log first (see per-tab routing).

In-progress resume banner

SHELL · resume

"Workout in progress" pill

When a session was minimised (not finished), an accent banner floats just above the tab bar — only on the Log tab — so the user can jump back in.

Layout & copy
  • Rendered by combined/log.jsx, gated on props.inProgress. Absolutely positioned: bottom: 96 · left: 14 · right: 14, sitting in the gap above the tab bar.
  • Solid --accent background, --r-lg radius. A bolt icon (20px), then a two-line block — "Workout in progress" (15px / 800) over "Tap to resume your session" (13px / 85% opacity) — then a translucent "Resume" chip (white 20% fill) pinned right.
  • The whole banner is one <button>; tapping anywhere on it resumes.
Visible

inProgress === true on the Log tab — a workout was minimised and not yet finished/discarded.

Hidden

inProgress === false, OR the user is on any tab other than Log, OR the overlay is currently open.

Prototype nuance → native intentIn the prototype, tapping the banner calls onStartWorkout(meta.name||'Workout', false) — it re-launches the seeded overview rather than restoring the exact live set-by-set state (the prototype doesn't snapshot in-progress reps). In the native build, "Resume" must restore the actual in-progress session — same exercises, logged sets, elapsed timer and rest state — not start a fresh one. Treat the prototype's re-launch as a stand-in for "re-open the live session."

The tab bar — TabBarEl

Defined in combined/ui.jsx, exported as window.AuraTabBar, and rendered by every tab (Log, Plan, Progress, Profile each drop one in at the bottom of their tree). It is a floating glass pill holding four tab buttons, with a separate circular FAB to its right, absolutely positioned over the screen above the home indicator.

Local state

StateInitDrives
fabOpenfalseFAB menu open/closed.
collapsedfalseCompact vs full height (scroll-driven).
hoverIdxnullPointer-hovered tab index (desktop preview).
swipeProg0Live swipe offset, −1…1, scaled to a quarter pill-width.
draggingfalseTrue during a touch drag → disables indicator transition.

Two static config arrays:

  • ALL_TABS = [log, Log, log] · [plan, Plan, dumbbell] · [progress, Progress, chart] · [profile, Profile, person] — each is [key, label, icon]. Order is fixed and defines swipe order + indicator index.
  • ACTIONS (FAB) = play-c / Start Workout / accent / workout, log / Log Measurements / blue / measure, medal / Progress Photo / green / photo.

Glass & structure

  • Outer bar: position:absolute; bottom:0, full width, flex-centered, height: 96px (66 collapsed), z-index:20, bottom padding 38px (30 collapsed) for the home-indicator safe area. Background transparent — the glass is on the pill, not the bar.
  • Glass pill: flex:1, border-radius:999px, background color-mix(in oklab, var(--text) 5%, transparent), backdrop-filter: blur(40px) saturate(2.5), hairline border color-mix(--text 11%), shadow 0 4px 32px rgba(0,0,0,.14) + inner top highlight inset 0 1px 0 rgba(255,255,255,.07), overflow:hidden. Max-width calc(100% − 62px) (leaves room for the FAB), shrinking to 72% collapsed.
  • Four tab buttons: each flex:1, column layout, 22px icon over a 10px label. Colors: active = #fff (over the indicator) · hovered-inactive = rgba(255,255,255,.7) · idle = var(--text-3). transition: color .2s. Clicking closes the FAB then calls onTab.
  • FAB circle: separate from the pill, to its right. 46×46 (34 collapsed), border-radius:50%, blur(40px). See the FAB section for open/closed styling.
EdgeEach tab renders its own copy of TabBarEl — there is no single persistent bar. Local state (collapsed, fabOpen, swipe) therefore resets when you switch tabs. In the native build, prefer one persistent tab bar with shared collapse state, but preserve the per-tab onAction routing differences documented below.

Single sliding indicator

TAB · indicator

One accent pill that slides

There is exactly one accent pill behind the icons. It translates horizontally to sit under the active tab — it never fades in/out per-tab.

Geometry
  • Absolutely positioned inside the glass pill: top:4 · bottom:4, background: var(--accent), border-radius:999px, glow shadow 0 2px 14px color-mix(--accent 50%), pointer-events:none.
  • Left = calc(4px + targetIdx × (100% − 8px) / 4); width = calc((100% − 8px) / 4) — i.e. exactly one quarter of the inner track, inset 4px each side.
  • targetIdx = hoverIdx if hovering, else clamp(tabIdx + swipeProg, 0, 3). So the pill previews where you're heading — it follows the finger mid-swipe and the cursor on hover, not just the committed tab.
Transition (depends on interaction)
StatetransitionWhy
dragging === truenonePill tracks the finger 1:1, no lag.
hoverIdx !== nullleft .18sSnappy desktop hover preview.
committed (tap / swipe release)left .32sSmooth settle to the new tab.

All three use the same cubic-bezier (.4, 0, .2, 1).

Hover preview (desktop / pointer)

Each tab button sets hoverIdx on mouseenter and clears it on mouseleave. While hovering, the indicator slides under the hovered tab and that tab's label brightens to 70% white — a pointer affordance only. On touch devices this never fires, so it's purely additive; safe to drop on a native build if the platform has no hover state.

Collapse on scroll

TAB · collapse

Shrinks when you scroll down

To stay out of the way while reading, the bar collapses on downward scroll and re-expands on upward scroll.

The signal
  • A global listener in Aura App.html captures scroll events on any element with class .screen-body (capture phase, so it catches every scroll container).
  • It assigns each scroller a data-scrollId, tracks last scrollTop per id, and — debounced via requestAnimationFrame with a 4px threshold — dispatches a window CustomEvent('aura:scroll', { detail:{ dir } }) where dir is 'up' or 'down'. Movements under 4px emit nothing.
  • Every TabBarEl subscribes on mount and sets collapsed = (dir === 'down'). (Subscribes in a useEffect, unsubscribes on unmount.)

NativeMap to the platform's "hide/shrink on scroll" tab-bar behaviour. The key feel to preserve: it reacts to direction, not absolute position — scrolling up even one notch re-expands immediately.

Swipe between tabs

TAB · swipe

Horizontal drag on the bar switches tab

A horizontal touch-drag on the tab bar itself moves between adjacent tabs, with the indicator previewing the destination live under the finger.

Gesture math (exact)
  • touchStart: store touchX = clientX, set dragging=true, swipeProg=0.
  • touchMove: dx = clientX − touchX. Compute prog = clamp(dx / (pillWidth/4), −1, 1) (pillWidth from the live offsetWidth, fallback 280). Apply swipeProg = −prog only if the move is toward a tab that exists: (prog<0 && tabIdx<3) || (prog>0 && tabIdx>0). At the ends, the indicator won't preview past the boundary.
  • touchEnd: dx = changedTouches[0].clientX − touchX; reset dragging/swipeProg. If |dx| < 35px → snap back, no change. Else: dx<0 && tabIdx<3 → next tab; dx>0 && tabIdx>0 → previous tab.
Commit

|dx| ≥ 35px toward an existing neighbour → onTab to it, indicator settles (.32s).

Snap-back

|dx| < 35px, or drag toward a boundary (already on first/last) → swipeProg resets to 0, indicator returns.

Boundary guardBoth the live preview and the commit are guarded so you can never swipe past the ends — left of Log or right of Profile does nothing and shows no preview drift.
NativeMap to a standard paged/swipe tab container. Two details worth keeping for the same tactile feel: the 35px commit threshold, and the indicator-follows-finger preview (driven by swipeProg) during the drag.

FAB quick-action menu

SHELL · FAB

Plus button → three quick actions

The circular accent + to the right of the pill toggles a stack of frosted-glass action chips above the bar.

The three actions
IconLabelKeyChip colour
play-cStart Workoutworkout--accent
logLog Measurementsmeasure--blue
medalProgress Photophoto--green
Open / close behaviour
  • Tapping the FAB toggles fabOpen. The glyph rotates 0° → 45° (+ becomes ×) over .2s, and the circle swaps from solid accent (with accent glow shadow) to a frosted neutral color-mix(--bg 55%) with a soft shadow.
  • Open renders a full-screen invisible catcher (position:absolute; inset:0; z-index:80) that closes the menu on any outside tap.
  • Chips stack bottom-up at bottom: barH + 10, right-aligned, 10px gap, z-index:90. Each is a glass pill (same blur/saturate as the bar) with a coloured 32px icon circle + label, animating in with fabItemIn (rise + fade) on a staggered delay (0.06 + i × 0.05)s so they cascade.
  • Selecting a chip closes the menu (setFabOpen(false)) then fires onAction(key). Tapping any tab button also closes the menu first.
Closed

Accent circle, + glyph, accent glow shadow. No chips, no catcher.

Open

Frosted circle, rotated ×, three cascading chips above, scrim catcher active.

Per-tab FAB routing (don't paper over this)

Inconsistency in the prototypeEach tab passes its own onAction handler, and they are not consistent — on Plan and Progress, all three actions just go to Log. The table below is the literal prototype wiring and the destination each action should reach. Build to the intent column.

The prototype's tab-to-Log fallbacks are shortcuts, not the spec.

Z-index map

The shell stacks several absolutely-positioned layers over each screen body. Keep this order so the catcher, chips and bar interact correctly:

z-indexLayerNotes
20Tab bar (pill + FAB)Always above the scrolling screen body.
80FAB scrim catcherFull-screen invisible; only present while fabOpen. Sits below the chips so chips stay tappable.
90FAB chip stackAbove the catcher and the bar.

The resume banner sits in normal flow above the bar at bottom:96 (no special z-index — it's just below the bar's layer). The Active Workout overlay isn't layered at all — it replaces the entire tab tree, so nothing here applies while it's open.

Scenario checklist

Your QA pass for this chapter. Each line is a discrete behaviour to verify.

  • App boots on Log; theme/week-start/log-stat restored from storage (or defaults on a fresh install).
  • Tapping each tab switches content; indicator slides smoothly (.32s) under the new tab.
  • Swiping the bar >35px moves one tab over; <35px snaps back; swiping past Log/Profile does nothing.
  • During a swipe, the indicator follows the finger and previews the destination; it stops at the boundaries.
  • Scrolling a screen down collapses the bar (icon-only, smaller FAB); scrolling up re-expands it.
  • FAB opens with cascading chips + scrim; outside tap or tab tap closes it; glyph rotates to ×.
  • Each FAB action reaches its intended destination from every tab (see routing table).
  • Starting a workout (any of the 3 entry points) opens the full-screen overlay with no tab bar.
  • Minimise → lands on Log with the resume banner; Exit/Finish/Discard → lands on Log with no banner.
  • Resume banner appears only on Log, only while a session is minimised; tapping it re-opens the live session.
  • Dark mode applies everywhere — including Plan/Workout sub-views that would otherwise force light.
  • Changing week-start or log-stat in Profile is reflected in Log/Progress and survives a reload.
