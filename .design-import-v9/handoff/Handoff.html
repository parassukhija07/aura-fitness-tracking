Developer Handover

Aura Fitness — build spec

A complete, screen-by-screen, scenario-by-scenario specification for rebuilding Aura natively. This document is written so that nothing built into the prototype gets lost in translation — every state, sheet, timer, animation and edge case is called out explicitly.

What this is

Aura is a personal workout-tracking app: it serves training programs and runs an advanced live workout tracker. The source of truth is the interactive prototype Aura App.html — a fully working React app (four tabs + a full-screen Active Workout flow). The HTML/JS files are design references, not production code. Your job is to recreate the behaviour in the target codebase using its own patterns.

This handover is split into 8 chapters / phases. Build them roughly in order — Foundation and Shell first, then one tab at a time. Each chapter documents every screen as: purpose → layout → every interactive element → all states/scenarios → edge cases.

FidelityHigh. Final colours, type, spacing, motion and interaction are all specified. styles/aura.css is the single source of truth for design tokens — read it before writing any UI.

How to use this document

  • Read Chapter 1 · Foundation first — it defines the tokens, shared components and the global state/persistence model that every screen depends on.
  • Work tab-by-tab. Within a chapter, build the screens top to bottom; the Scenarios blocks are your test checklist.
  • Treat every SCREEN-ID badge as a discrete unit of work. The IDs are stable references you can use in commits, tickets and QA.
  • Anywhere you see a EDGE or warn callout, that's a behaviour that is easy to miss — implement it deliberately.

Target stack

The design follows Apple HIG (large titles, grouped lists, bottom sheets, segmented controls, a floating tab bar). For iOS, SwiftUI is recommended. If a cross-platform stack is already in place (React Native / Expo, Flutter), rebuild there with that stack's native components. Do not ship the prototype HTML.

  • Map icons to SF Symbols where equivalents exist (the prototype ships a custom 24px stroke set, weight 1.7, round caps — see Chapter 1).
  • Substitute the system font (SF Pro) for Plus Jakarta Sans if preferred — layouts tolerate it. Keep tabular numerals on all stats/timers.
  • All imagery is placeholder (striped boxes with monospace captions like "exercise demo", "body map"). Replace with real assets — see each chapter's asset notes.

App map

One root component (AuraRoot) owns the active tab and a handful of cross-tab settings (theme, week-start, log-stat). Four tabs render inside an iPhone 393×852 frame; the Active Workout flow is a full-screen overlay that takes over the whole device when a session runs.
